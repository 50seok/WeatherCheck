# 🔧 트러블슈팅 — 디스코드 봇 무응답 근본원인 (온보딩 403 → on_message 전체 중단)

**날짜**: 2026-07-04 밤 (PR#3, 커밋 `4ac440b` 머지)

**상황**: 사용자가 봇에게 메시지를 보내도(예: 알림 시각 "2015") 아무 응답이 없음. 겉보기엔 시각 인식 실패처럼 보였지만 실제로는 봇이 메시지 자체를 처리하지 못하고 있었음.

**증상 (표면)**:
1. "알림설정 2019"처럼 조사(에/로) 없이 명령어+시각이 붙은 압축 표현을 `classify_message`가 NONE으로 오분류
2. 메시지 전체가 4자리 숫자뿐인 경우("2015" 등) LLM이 시각인지 일반 숫자 언급인지 판단을 못 해 시각 추출 실패

**근본원인**: 채널 최초 사용자를 위한 온보딩 안내 패널 자동 전송 기능(신규 추가분)이, 봇 권한이 부족한 채널에서 Discord API 403 Forbidden을 반환할 때 예외 처리가 없었음 → 이 예외가 `on_message` 이벤트 핸들러 전체를 중단시킴 → 이후 사용자가 보낸 실제 메시지가 분류·응답 단계까지 아예 도달하지 못하고 무응답으로 보였던 것. 표면 증상 1·2번은 부수적 오분류였을 뿐, 진짜 원인은 온보딩 코드의 미흡한 예외 처리였음.

**해결**:
- 온보딩 패널 전송을 `try/except`로 감싸 403 등 실패해도 `on_message` 처리가 계속되도록 수정 (`src/discord_chat_bot.py`)
- 표면 증상 1·2도 각각 프롬프트 예시 추가 / LLM 호출 전 결정적 정규식으로 유효 시각(0000~2359) 선(先)체크하도록 함께 수정
- 진단용 로그 1줄 추가(`[on_message] '메시지' -> intent=... detail=...`) — 유사 무응답 리포트 시 원인 파악 속도 개선

**재발 방지**:
- **이벤트 핸들러 안에서 실행되는 부가 기능(로깅, 알림, 온보딩 등)은 반드시 자체 예외 처리를 갖춰야 함** — 부가 기능 하나의 실패가 핵심 로직(메시지 분류·응답) 전체를 막으면 안 됨. 핵심 로직과 부가 기능은 실패 격리(fail isolation)가 원칙.
- "무응답" 버그 리포트를 받으면 먼저 분류 로직을 의심하기 쉬운데, **핸들러 진입 자체가 막혔는지**(예외로 인한 조기 종료)부터 로그로 확인하는 게 우선순위. 이번 건도 표면 증상만 보고 분류 프롬프트만 고쳤다면 근본원인을 놓칠 뻔했음.
