# 📋 PRD — 나만의 출근 비서 (Personal Commute Assistant)

> **한 줄 컨셉:** 기상청은 *날씨*를 알려주지만, 내 비서는 *내가 뭘 해야 할지*를 알려준다.
>
> 작성일: 2026-06-18 · 상태: 기획 · 학습 프로젝트 (ML → DL → LLM)

---

## 1. 배경 (Why)

- 날씨 앱은 많지만 대부분 "정보 표시"에서 멈춤 (기온 18도, 강수 70%).
- 사용자가 진짜 원하는 건 숫자가 아니라 **행동 지침** — "겉옷 챙겨? 우산 들어? 몇 시에 나가?"
- 공개 날씨 예측은 기상청이 더 잘함 → **차별점은 "개인화"**: 나의 체감·루트·일정에 맞춘 조언.

## 2. 문제 정의 (Problem)

| 기존 | 문제점 |
|---|---|
| 날씨 앱 | 숫자만 줌, 해석은 내 몫 |
| 기상청 예보 | 정확하지만 "그래서 뭘 해야 하나" 없음 |
| 일반 날씨봇 | 누구에게나 똑같음 (내 체감/루트 무시) |

→ **결론:** "공개 날씨 예측"이 아니라 **"나의 출근 결정 예측"** 으로 풀어야 의미 있고 차별화됨.

## 3. 목표 (Goal)

- **핵심 목표:** 출근 전, 나에게 맞춘 행동 지침을 폰 알림으로 자동 발송.
- **학습 목표:** 하나의 테마로 ML → DL → LLM 풀스택 경험.
- **차별화 목표:** "나만의 데이터"로 겹칠 수 없는 프로젝트 만들기.

## 4. 타겟 사용자 (Who)

- 1차: **나 자신** (출퇴근하는 직장인)
- 확장(니치): 🚲 자전거 출퇴근족 / 🏍️ 배달 라이더 / 😷 미세먼지 민감자 / 👶 아이 등하교 부모
- 니치를 좁힐수록 차별화 ↑ (예: "자전거 출퇴근족 전용 날씨+노면+미세먼지 비서")

---

## 5. 차별화 전략 (Differentiation) ⭐

> 겉모습(날씨 앱)은 흔하다. **깊이(ML→DL→LLM)** 와 **개인화** 로 차별화한다.

### ① 개인 체감 학습 (가장 강력)
```
알림 후 → "오늘 추웠어 / 딱 좋았어" 피드백 버튼
→ 모델이 "나의 체감온도" 학습
→ 남들은 18도에 안 입어도, 나는 겉옷 입는 걸 학습
```
- 효과: ML이 의미 있어지고(공개예보 재탕 X), 기술적으로 차별화됨.
- **피드백 루프**가 이 프로젝트의 시그니처.

### ② 니치 타겟
- 대상을 좁혀 "모두의 날씨" → "특정 상황 전용 비서"로.

### ③ 내 루틴 결합
- 캘린더 연동("오늘 외근 → 우산+일찍"), 내 출근 시간만 예보.

### 차별화 한 줄
> "기상청은 날씨를 알려주지만, 내 비서는 **내가 뭘 해야 할지**를 알려준다."

## 6. 핵심 기능 (Features)

| 기능 | 설명 | 단계 |
|---|---|---|
| 날씨 예측 | 내일 기온/강수 예측 | Phase 1 (ML) |
| 시계열 예보 | 지난 7일 → 향후 3일 정교 예측 | Phase 2 (DL) |
| 자연어 브리핑 | 숫자 → "비 오니 우산" 말로 변환 | Phase 3 (LLM) |
| 교통 결합 | 대중교통 혼잡 / 자차 정체 안내 | Phase 3 |
| 폰 알림 | 매일 아침 자동 발송 | Phase 3 |
| 개인 체감 학습 | 피드백으로 나만의 기준 학습 | 확장 |
| 📊 발표자료 | 매 단계 5장(문제→데이터→방법→결과→배운점) | 전 단계 |
| 🚀 Streamlit 배포 | 매 단계 데모 앱, 점진 확장 | 전 단계 |

### 조언 로직
| 조건 | 알림 문구 |
|---|---|
| 강수확률 높음 | ☔ 우산 챙기세요 |
| 일교차 큼 | 🌡️ 겉옷 챙기세요 |
| 미세먼지 나쁨 | 😷 마스크 챙기세요 |
| 대중교통 혼잡 | 🚇 10분 일찍 나가세요 |
| 자차 + 정체 | 🚗 우회도로 추천, 30분 소요 |

## 7. 단계별 범위 (Scope / 단계별 MVP)

### Phase 1 — ML (날씨 예측)
- **데이터:** 기상청 과거 날씨 CSV (data.kma.go.kr)
- **입력:** 오늘 기온·습도·기압·어제 강수량 → **정답:** 내일 기온(회귀)/비여부(분류)
- **모델:** LinearRegression → RandomForest → XGBoost
- **기술:** scikit-learn
- **📊 발표:** EDA 시각화 + 모델별 정확도 비교 차트 (5장)
- **🚀 배포:** Streamlit 데모 (슬라이더 입력 → 내일 날씨 예측)
- **산출물:** 코드 + 발표자료 + Streamlit 앱
- **완료 기준:** 예측 + 정확도(MAE/Accuracy) 산출, EDA 완료, **Streamlit 데모 작동**

### Phase 2 — DL (시계열)
- **왜:** 날씨는 순서가 중요 → LSTM이 순서를 기억
- **입력:** 지난 7일 시퀀스 → **출력:** 향후 1~3일
- **기술:** TensorFlow/Keras (또는 PyTorch)
- **전환:** CSV → 실시간 기상청 API
- **📊 발표:** 학습곡선 + "ML vs DL 정확도 비교" + 예측 그래프 (5장)
- **🚀 배포:** Streamlit 확장 (Phase 1 앱에 며칠치 예보 그래프 + 실시간 API)
- **산출물:** 코드 + 모델 저장(.h5) + 발표자료 + 업그레이드된 Streamlit 앱
- **완료 기준:** ML 대비 정확도 비교, 다일 예보, **Streamlit에서 ML/DL 나란히 비교**

### Phase 3 — LLM (통합 비서)
- **재료:** DL 예측(날씨) + 교통 API(혼잡/정체) + 조언 규칙
- **동작:** 아침 7시 스케줄러 → DL예측 + 교통조회 → LLM 브리핑 생성 → 폰 알림
- **기술:** Claude/OpenAI API + 스케줄러(cron) + 알림봇(디스코드)
- **📊 발표:** 아키텍처 다이어그램 + 실제 알림 데모 영상/스크린샷 (5장)
- **🚀 배포:** Streamlit 대시보드(오늘 브리핑·예보·교통 한 화면) + 디스코드 봇 실서비스
- **산출물:** 코드 + 발표자료 + Streamlit 대시보드 + 작동하는 알림 봇
- **완료 기준:** 매일 아침 폰으로 자연어 출근 브리핑 자동 도착, **Streamlit 대시보드 배포**

## 8. 기술 스택

| 영역 | 도구 |
|---|---|
| 분석/ML | Python, pandas, numpy, scikit-learn, xgboost |
| 시각화 | matplotlib, seaborn |
| DL | TensorFlow/Keras (or PyTorch) |
| LLM | Claude API / OpenAI API |
| 자동화 | cron 스케줄러 |
| 알림 | 디스코드 봇 (가장 쉬움) |
| **배포** | **Streamlit** (전 단계 통일) → Streamlit Community Cloud(무료) or OCI 서버 |
| **발표** | 마크다운→슬라이드(Marp) or 주피터 노트북, 단계별 5장 |
| 데이터 | 기상청(data.kma.go.kr / API), 교통 API(공공데이터포털) |
